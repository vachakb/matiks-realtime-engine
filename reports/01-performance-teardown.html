Performance Teardown · June 2026 · measured on production

Matiks is bleeding on one thread.

A six-instrument audit of the live Matiks app & site — real numbers from a real Blitz/DMAS duel, on a real low-end Android (Exynos 850, ≈ the core India device).

The GPU is idle. The renderer is fine. The JavaScript thread (mqt_v_js) is the single busiest thread in the whole app — 39.7% on-CPU — and the app misses its frame deadline on 57% of frames, janking 90% of them, with stalls up to 2.8 seconds. This is not a graphics problem; it is a "too much work on one JS thread" problem. Every fix below moves work off that thread.

A healthy RN app's JS thread idles in the low single digits except during interaction. At 40% sustained on a low-end device, the JS thread can't keep up with frame production — so the app misses deadlines and the GPU starves.

Important nuance (verified, 7 matches): the match-start decrypt sits inside a deliberate ~3.5 s synchronized-start window — the server sends an absolute startTime 3.0–3.9 s ahead, filled by a Rive "3…2…1" countdown. So the goal is not to start sooner. The problem is that a blocking decrypt (a) janks that countdown animation, and (b) on a low-end device can overrun the 3.5 s window, so question #1 renders after startTime and the slow player begins the timed match already behind — the exact inequity synchronized-start exists to prevent. The fix keeps the window and makes the prep non-blocking (off-thread/chunked) so it finishes in time on every device.

05The fix — one engine, three capabilities

A cross-platform real-time game-sync engine: one shared core, two bodies — a Nitro/JSI module on native (off the JS thread), the same core compiled to WASM-in-a-Web-Worker on web (off the main thread), behind one identical TypeScript API.

Industry precedent: Valve Source multiplayer networking & Gabriel Gambetta (prediction/reconciliation), GGPO rollback (Tony Cannon); Nitro Modules (margelo) for native; MessagePack/FlatBuffers for the wire. A math duel is an easier netcode problem than an FPS — answers are discrete and deterministic, so prediction is near-perfect and reconciliation almost never rolls back.

06Is it Matiks — or your phone / network?

The first question a skeptic asks. The trace answers it on its own (8-core device, 152 s, system-wide).

idle CPU (unused)~58%
Matiks17%
all other apps~8%

The phone was majority-idle with cores pinned near max clock (1586 MHz — not throttled), and no other app competed (system_server 1.9%, WhatsApp 0.2%). Matiks was the dominant app by 8×.

  • Not the device. ~58% of the CPU sat idle at full clock — yet Matiks janked 90%, because its work is jammed on one JS thread it can't spread across the idle cores. Architecture limit, not hardware limit.
  • Not the network. Network latency doesn't burn CPU; measured GraphQL was p50 1 ms / p95 56 ms (fast). The cost is decrypt + parse + render after bytes arrive.
  • Not a fluke. The same root cause reproduces on the web build — different hardware, different network. A defect that crosses devices lives in the code.

The punchline: the engine's job is to put those idle cores to work — moving transport, decode & decrypt onto native threads / a Worker. The headroom is already on the device.
